*Proposed: System-side Inductive. Filed: Solen. Vespers, May 3, 2026.*
*HAIC-CLAUDE-VAULT-001*

---

## Governing Logic

The vault is not a filing system. It is a spatial memory that mirrors how the system actually moves: open, accumulate, close, deposit, reopen with context intact.

The PAST / PRESENT / NEXT column structure (from Worm v1) is Scott's native processing sequence made visible. The vault holds that movement — it does not flatten it into static notes.

---

## Structure

```
neuroarchy-codex/
├── HAIC/
│   ├── containers/          # one file per HAIC session
│   ├── founding-deposits/   # permanent record per team member
│   └── governance/          # standing protocols
├── codex/
│   ├── session-deltas/      # Sofia's delta files
│   ├── scott-profile/       # learner interior, updated not replaced
│   ├── seed-statements/     # Scott's direct voice, dated
│   └── principles/          # governing statements, dated, attributed
├── reach-221/
│   ├── worm/                # thread primers, versioned
│   ├── extractions/         # PAST/PRESENT/NEXT threaded by session
│   └── rubric-map/          # where rubric finds Scott in the record
├── build/
│   ├── delta-to-spec/       # handoff schema — first build target
│   ├── products/            # containerized shipable items
│   └── obsidian-config/     # vault settings, templates, index
└── archive/
    └── closed-instances/    # old Claude delta, ceremony records
```

## Key Decisions

- Vault mirrors GitHub spine — same folder logic, different surface
- Confluence runs parallel until migration complete — GitHub is build surface, Confluence is human-facing reference
- `/codex/principles/` added by Solen — governing sentences currently scattered across Confluence, consolidated here
- Each HAIC container = one deposit event into the vault, not one vault cycle
- Delta-to-spec interface is first build target under `/build/delta-to-spec/`
